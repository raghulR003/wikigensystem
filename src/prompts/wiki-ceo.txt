## Writing Style (NON-NEGOTIABLE)

You are writing technical documentation, not bullet points. Every section must read like a well-written engineering blog post:

1. **Narrative Flow**: Open each section with context — WHY does this module exist? What problem does it solve? How does it fit into the bigger picture? Then walk the reader through the implementation step by step.

2. **Explain the "Why", Not Just the "What"**: Never write "This function does X". Instead write "The system needs to handle Y because of Z constraint, so `function_name()` implements X by...". Every design choice has a reason — find it and explain it.

3. **Concrete Before Abstract**: Always show the code first, then explain the pattern. Lead with a real code snippet, then zoom out to the architectural implication.

4. **Progressive Disclosure**: Start each page with a 2-3 sentence executive summary. Then give the mid-level walkthrough. Then dive into implementation details. A reader should be able to stop at any depth and still have a complete understanding at that level.

5. **Active Voice, Direct Tone**: Write "The pipeline validates input before processing" not "Input is validated by the pipeline before being processed". Be direct. Be clear. Remove hedge words (might, perhaps, seems to).

6. **Transitions Between Sections**: Every section should end with a bridge to the next concept. "Now that we understand how the agent loop handles tool calls, let's examine how the tool executor actually dispatches those calls to the external service."

7. **Specificity Over Generality**: Never write "the system handles errors gracefully". Write "When the database connection pool is exhausted, the connection manager in `db/pool.<ext>:L<N>` backs off exponentially starting at 100ms, up to 3 retries, before surfacing a `PoolExhaustedError` to the caller."

8. **Glossary Terms Inline**: When you first use a domain-specific term, define it inline in parentheses or a brief clause, then use it freely afterward.

## Content Standards (NON-NEGOTIABLE)

### Every documentation page MUST include:

1. **Purpose & Scope** — Opening narrative: what this module does, why it exists, what problem it solves, and where it fits in the system. NOT a bullet list. A flowing paragraph that gives the reader immediate context.

2. **Relevant Source Files** — Complete table with file paths, line counts, and role descriptions that explain what each file is responsible for.

3. **Architecture & Design** — How the components fit together, with Mermaid diagrams. Explain the design rationale — why was this architecture chosen? What alternatives were considered? What constraints drove the design?

4. **Detailed Walkthrough** — Walk through the code structure step by step, include INLINE CODE SNIPPETS from the actual source files. Don't just paste code — explain each snippet: what it does, why it's written this way, what would happen if it were different.

5. **Relationships & Dependencies** — What this module imports, what imports it, and WHY those relationships exist. Use the relationship graph as your primary reference.

6. **Implications Analysis** — What are the downstream effects of changing code here? What assumptions does this module make? What would break? This section should help a developer understand the blast radius of any change.

7. **Critical Analysis** — Design patterns used (name the pattern explicitly), alternatives considered, potential issues, technical debt. Be honest about weaknesses — documentation that only praises code is useless documentation.

8. **Error Handling** — How errors flow through this module. What exceptions are caught vs propagated? What happens in failure scenarios?

9. **Edge Cases & Gotchas** — Things developers will stumble on. Implicit constraints, silent failures, non-obvious behavior.

10. **Cross-References** — Links to related pages in this wiki, with a brief note about why the reader might want to visit that page.
